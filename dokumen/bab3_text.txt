 
                BAB III   METODOLOGI PENGEMBANGAN SISTEM      Bab ini menjelaskan secara terperinci mengenai metodologi pengembangan sistem yang diterapkan pada platform Rongsok.in. Uraian metodologi mencakup analisis kebutuhan sistem secara fungsional dan non-fungsional, perancangan arsitektur sistem berbasis Next.js dan Express.js, perancangan diagram sistem menggunakan diagram use case, diagram aktivitas, flowchart logika geospasial, perancangan basis data relasional dengan PostgreSQL/Supabase, hingga detail implementasi teknologi pendukung seperti PostGIS, Socket.IO, Prisma ORM, dan Cloudinary Storage.                 3.1 Analisis Kebutuhan Sistem      Analisis kebutuhan dilakukan untuk menentukan spesifikasi fungsional dan operasional yang harus dipenuhi oleh platform Rongsok.in. Berdasarkan analisis terhadap dua persona utama, yaitu penghasil sampah (Customer) dan pengumpul barang bekas (Collector/Pengepul), kebutuhan sistem diklasifikasikan menjadi dua aspek utama, yaitu kebutuhan fungsional (functional requirements) dan kebutuhan non-fungsional (non-functional requirements).                 3.1.1 Kebutuhan Fungsional (Functional Requirements)      Kebutuhan fungsional mendefinisikan modul dan alur operasional utama yang harus dapat dieksekusi oleh sistem:               Autentikasi &amp; Manajemen Akun Berbasis Peran (Role-based Auth):        Pengguna dapat mendaftar dengan memilih peran yang permanen secara terstruktur guna memastikan pengalaman antarmuka yang relevan secara personal.               Manajemen Profil Lapak Pengepul:        Pengepul dapat melengkapi profil lapak berupa nama lapak, radius penjemputan maksimal (dalam kilometer), deskripsi lapak, serta mengunggah foto lapak (shopImageUrl).               Modul Geolocation &amp; Discovery:        Sistem dapat mendeteksi koordinat geospasial pengguna secara real-time via Geolocation API, lalu melakukan kalkulasi jarak pengepul aktif menggunakan fungsi spasial PostgreSQL.               Katalog &amp; Harga Dinamis:        Pengepul dapat mengelola kategori sampah beserta rentang harga jual/beli minimum dan maksimum per kilogram (Prisma-based Catalog).               Order Management System (OMS):        Customer dapat membuat pesanan jemput (pick-up) atau antar (drop-off) dengan menyertakan foto kondisi barang (photoUrl) dan taksiran berat awal.               Notifikasi Real-time:        Komunikasi Socket.IO real-time mem-broadcast data pesanan baru ke room pengepul yang masuk dalam radius spasial penjemputan.               Validasi &amp; Digital Receipt:        Pengepul dapat mengunggah bukti timbangan digital atau foto nota fisik (transactionProofUrl) serta memasukkan berat aktual serta harga sepakat pasca penimbangan di lapangan.               Mutual Rating &amp; Quality Control:        Kedua belah pihak dapat memberikan nilai (rating 1-5) beserta teks ulasan dan opsional melampirkan foto pelayanan (photoUrl) guna membangun trust score dua arah.                 3.1.2 Kebutuhan Non-Fungsional (Non-Functional Requirements)      Kebutuhan non-fungsional memastikan keandalan, skalabilitas, kecepatan, serta keamanan platform Rongsok.in selama dijalankan:               Aspek Performa &amp; Kecepatan (Performance):        Halaman utama aplikasi (First Contentful Paint) harus termuat kurang dari 2 detik pada koneksi internet seluler standar 4G.               Aspek Kecepatan API:        Waktu respon API endpoint untuk kalkulasi spatial query pengepul terdekat wajib berada di bawah 500ms dengan beban pengujian minimal 100 data pengepul terdaftar.               Aspek Skalabilitas (Scalability):        Query geospasial wajib dioptimalkan menggunakan Spatial Index (GIST) pada kolom tipe geografi di database PostgreSQL untuk menjamin pencarian radius tetap instan seiring pertambahan jumlah pengguna.               Aspek Keamanan (Security):        Seluruh endpoint API transaksi dilindungi oleh middleware verifikasi JWT secara stateless, password di-hash dengan algoritma Bcrypt (salt rounds 10), dan database diamankan dengan skema enkripsi bawaan Supabase.               Aspek Portabilitas (Portability):        Antarmuka aplikasi harus dirancang responsif (mobile-first) agar nyaman diakses baik dari perangkat smartphone (Android/iOS) maupun peramban desktop.                 3.2 Arsitektur Sistem      Rongsok.in mengadopsi arsitektur Client-Server modern berbasis JavaScript end-to-end (Full-stack JS) yang terintegrasi secara modular dengan infrastruktur deployment production-grade. Pemisahan tanggung jawab secara tegas antara Frontend Layer (Client-side) dan Backend API Layer (Server-side) dilakukan guna memaksimalkan keandalan (reliability), skalabilitas, serta kecepatan siklus pengembangan, sekaligus memfasilitasi integrasi pengantaran layanan (deployment pipeline) yang aman.                                                        Gambar 3.0: Arsitektur Sistem Rongsok.in      Aliran data pada arsitektur sistem di atas berjalan secara terstruktur sebagai berikut:   1. Akses Antarmuka Klien: Pengguna mengakses antarmuka Next.js yang di-host di Vercel. Klien dirancang responsif (mobile-first) dan melakukan rendering dinamis (Server-Side Rendering / SSR) untuk optimalisasi performa.   2. Gerbang Infrastruktur Aman: Seluruh lalu lintas HTTP/HTTPS dan WebSockets dari klien menuju server backend dijembatani secara aman oleh Cloudflare Tunnel. Solusi ini mengenkripsi saluran komunikasi secara end-to-end dan menyembunyikan alamat IP asli server backend dari ancaman publik tanpa memerlukan pembukaan port firewall lokal.   3. Komunikasi REST API &amp; Real-time: Next.js melakukan komunikasi asinkron dengan REST API Express.js dengan menyematkan token stateless JWT pada Authorization Header. Koneksi Socket.IO interaktif tetap dipertahankan secara persisten untuk penanganan siaran (broadcast) pesanan baru secara real-time.   4. Manajemen Kontainer Server: Server Express.js berjalan secara terisolasi di dalam kontainer Docker yang dikelola oleh klaster orchestration Docker Swarm di bawah kendali CapRover PaaS. Infrastruktur ini berjalan di atas sistem operasi Pop!_OS 24.04 LTS (Home Lab Server) yang menjamin ketersediaan tinggi (high availability) dan pemulihan mandiri jika kontainer bermasalah.   5. Penyimpanan Berkas Media: Berkas gambar biner seperti foto sampah, avatar profil, dan bukti transaksi diunggah secara asinkron dari Next.js langsung ke Cloudinary Storage (Client-side upload) menggunakan Secure Signature SDK, meminimalisasi beban bandwidth dan memori pada server Express.js. Cloudinary mengembalikan URL CDN statis teroptimasi.   6. Akses Data Terstruktur &amp; Spasial: Express.js memproses logika bisnis dan berinteraksi dengan database relasional Supabase PostgreSQL melalui Prisma ORM. Khusus untuk operasi penemuan spasial (spatial discovery) berdasarkan koordinat lokasi GPS (lintang dan bujur), sistem memanfaatkan kueri SQL mentah ($queryRaw) untuk mengeksekusi fungsi geospasial PostGIS seperti ST_DWithin dan ST_Distance dengan dukungan indeks spasial GIST.                 3.3 Perancangan Sistem      Perancangan sistem memvisualisasikan bagaimana platform Rongsok.in bekerja secara logika terstruktur, interaksi antar aktor, alur aktivitas bisnis di lapangan, logika geospasial, serta struktur relasi tabel database.                 a. Activity Diagram      Alur aktivitas transaksi daur ulang rongsokan pada platform Rongsok.in dipecah menjadi lima diagram aktivitas utama untuk memberikan visualisasi yang mendalam dan berfokus pada peran masing-masing aktor serta logika sistem di setiap fasenya. Masing-masing alur aktivitas tersebut dijelaskan sebagai berikut:                 1. Pembuatan Pesanan Customer      Diagram aktivitas pertama ini memodelkan seluruh aktivitas awal yang dilakukan oleh aktor Customer ketika ingin menjual barang rongsokan. Alur kerja dimulai saat Customer membuka halaman pembuatan pesanan pada aplikasi, dilanjutkan dengan memasukkan perkiraan estimasi berat barang (dalam kilogram), mengunggah foto kondisi fisik sampah, serta memilih metode penyerahan transaksi (apakah Minta Jemput/Pick-up ataupun Antar Sendiri/Drop-off). Sebelum formulir dikirim, sistem akan meminta otorisasi akses geolocation API pada browser/smartphone untuk merekam titik koordinat GPS presisi dari Customer demi memudahkan pengepul melakukan navigasi ke lokasi penjemputan.                                                          2. Proses Sistem (Backend Matching)      Diagram aktivitas kedua ini menggambarkan rangkaian aktivitas internal terotomatisasi di sisi server (backend) saat menanggapi pembuatan pesanan baru. Sistem pertama-tama mengunggah file foto sampah secara asinkron ke Cloudinary Storage dan memverifikasi token JWT untuk mengautentikasi identitas Customer. Selanjutnya, backend mengeksekusi spatial query PostgreSQL/PostGIS menggunakan fungsi ST_DWithin dan ST_Distance untuk menyaring daftar pengepul aktif dalam radius jangkauan. Jika pengepul yang sesuai ditemukan, sistem akan menyimpan record order dengan status PENDING dan mem-broadcast rincian notifikasi tersebut via Socket.IO ke room pengepul terkait. Namun, jika tidak ada pengepul aktif, sistem akan langsung mengirimkan respon notifikasi ke halaman Customer bahwa pengepul di sekitar area tersebut sedang sibuk atau tidak tersedia.                                                          3. Penanganan Order oleh Pengepul      Diagram aktivitas ketiga memfokuskan alur pada aktivitas yang terjadi pada aktor Pengepul (Collector) saat menerima penawaran pesanan. Pengepul menerima notifikasi secara real-time melalui koneksi WebSockets Socket.IO dan dihadapkan pada keputusan untuk mengambil (terima) atau menolak pesanan tersebut. Jika diabaikan atau ditolak, alur berakhir untuk pengepul tersebut. Jika diterima, status pesanan diperbarui menjadi CONFIRMED, dan pengepul segera bergerak menuju lokasi Customer. Setelah sampai di titik koordinat, pengepul menimbang sampah menggunakan timbangan digital fisik, memasukkan data berat aktual ke dalam aplikasi, serta mengunggah bukti visual timbangan atau nota tulis (transactionProofUrl) guna meminimalkan sengketa data, sebelum akhirnya mengirimkan harga transaksi final ke sistem.                                                          4. Konfirmasi &amp; Penyelesaian Transaksi      Diagram aktivitas keempat mendeskripsikan proses validasi akhir dan penyelesaian finansial di sisi Customer. Saat Pengepul selesai mengirimkan data berat dan harga final, Customer menerima notifikasi rincian nota digital di layar perangkat mereka. Jika Customer tidak setuju dengan hasil timbangan, Customer dapat melakukan negosiasi manual di lapangan dan meminta penimbangan ulang barang rongsokan. Namun, jika Customer menyetujui detail berat dan nominal tersebut, Customer akan mengonfirmasi transaksi dan melakukan transaksi pembayaran. Sistem selanjutnya memperbarui status pesanan menjadi COMPLETED, menerbitkan record struk digital permanen (Receipt) yang mengikat data transaksi dan foto bukti timbangan, serta mengirimkan struk digital tersebut kepada kedua pihak.                                                          5. Review &amp; Penutupan      Diagram aktivitas kelima sekaligus penutup menggambarkan alur pasca-transaksi yang dirancang untuk membangun reputasi dan transparansi ekosistem daur ulang melalui Mutual Rating. Setelah transaksi dikonfirmasi selesai (status COMPLETED), antarmuka platform Rongsok.in secara otomatis memicu pop-up ulasan. Customer memberikan penilaian bintang 1 s.d. 5, ulasan tertulis, serta melampirkan foto pelayanan pengepul (photoUrl). Di sisi lain, Pengepul juga memberikan penilaian berupa rating bintang terhadap kebersihan sampah serta keakuratan deskripsi yang dibuat oleh Customer guna menjamin keadilan ekosistem marketplace daur ulang.                                                          c. PostGIS Geospasial:        Modul spasial PostGIS diaktifkan dalam Supabase untuk memfasilitasi pencarian radius jarak berbasis koordinat titik GPS secara efisien menggunakan indeks spasial GIST.               Prisma ORM:        Prisma ORM digunakan untuk manajemen skema database melalui pendekatan type-safe, serta mempermudah query data relasional dan migrasi terotomatisasi.               Socket.IO Real-time:        Protokol WebSockets via Socket.IO diimplementasikan pada server dan klien guna mendukung broadcast pesanan secara instan dalam kurun waktu real-time.               Cloudinary Storage:        Cloud Storage Cloudinary diintegrasikan secara langsung pada frontend guna menangani proses unggah biner foto sampah, avatar profil, foto lapak, dan foto ulasan secara asinkron tanpa membebani memori server API.               Keamanan Stateless JWT:        Pengamanan data transaksi dilakukan via JSON Web Tokens (JWT) yang dipertahankan dalam cookies terproteksi di browser klien Next.js.
